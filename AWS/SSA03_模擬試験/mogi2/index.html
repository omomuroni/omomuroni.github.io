    Q1#

    このシナリオでは、オンプレミス環境とAWSネットワークの接続が求められており、その際に「プライマリの接続設定では低レイテンシーで可用性の高い安全な接続とすること」 「セカンダリの接続設定では、コスト最適にプライマリ接続障害時に冗長性を担保しつつも、トラフィック転送速度のパフォーマンス低下は許容されること」が要件となっています。

    「プライマリの接続設定では低レイテンシーで可用性の高い安全な接続とすること」に対しては、最も低レイテンシーで安全な接続が可能なAWS Direct Connectの専用線接続を 構成することで、オンプレミス環境からVPCのプライベートネットワークに直接疎通できるように構成します。

    AWS Direct Connect は50 Mbps から 100 Gbps まで速度を利用可能な、高性能な専用線接続を実施するサービスです。 専用線をDirect Connectロケーションという中継地点を介して、直接にAWS側と物理的に接続することで高速なネットワーク接続を可能にします。

    「セカンダリの接続設定では、コスト最適にプライマリ接続障害時に冗長性を担保しつつも、トラフィック転送速度のパフォーマンス低下は許容されること」に 対しては、AWS Direct Connect を二重化する方法と、AWS Direct Connect のオプションのAWS Site-to-Site VPNを追加して、 セカンダリの接続設定でIPsec (IP セキュリティ) を使用した安全な接続をする方法を選択できます。その中で、AWS Site-to-Site VPNを利用することで、 トラフィック転送速度のパフォーマンスはDirect Connectよりも低下しますが、コスト最適にプライマリ接続障害時に冗長性を担保することができます。

    このシナリオでは、リレーショナルデータベースを利用することが求められており、Amazon RDSまたはAmazon EC2インスタンスベースのリレーショナルデータベースが必要となります。 その中で、運用の手間をかけさせないという要件から、Amazon RDSが最適です。RDSのインスタンスタイプとストレージタイプの選択において、 IO性能が高いタイプを選択して高性能な処理が行えるように設定することで、ランダムI/O遅延を防ぐことができます。

    db.m5d は低レイテンシー、非常に高いランダム I/O パフォーマンス、高いシーケンシャル読み取りスループットに向けて最適化された汎用インスタンスクラスです。 このDBインスタンスにこのクラスを利用することで、ランダムI/O遅延が発生しないパフォーマンスを発揮できます。

    プロビジョンド IOPS ストレージは、低 I/O レイテンシーおよび一貫した I/O スループットが必要となる I/O 負荷の高いワークロード (特にデータベースワークロード) のニーズを満たすように設計されています。

    EC２インスタンスにインストールして構成するリレーショナルデータベースは、マネージド型で提供されないため、設定や運用に労力を要します。 したがって、なるべく運用負荷をかけないという要件を達成できません。

    このシナリオでは、インターネットを経由することなくAmazon EC2インスタンスからAmazon S3 APIを呼び出す必要があります。 そのためには、ゲートウェイエンドポイントを介して、Amazon EC2インスタンスからS3バケットに接続するように構成することが必要です。

    EC２インスタンスはVPC内のサービスであり、S3はリージョンに設置するVPC外のサービスとなります。 VPCからVPC外のサービスにアクセスしようとすると通常はインターネットを介して接続を実施します。 VPCエンドポイントを使用すると、インターネットを介さずに、AWSのネットワークバックボーンを利用して、 VPC内のサービスからVPC外のAWSサービスに接続することができます。

    これにより、インターネットを通さずにEC2インスタンスからS3バケットへのデータの登録と読取を実施することができます。

    ゲートウェイエンドポイントは、VPC にインターネットゲートウェイや NAT デバイスを必要とせずに、Amazon S3 および DynamoDB への信頼性の高い接続を提供します。 S3 や DynamoDB へ接続したいリソースが配置されている VPC に VPC エンドポイントを割り当て、ルートテーブルに ターゲットが VPC エンドポイントのルーティング を設定します。

    また、以前はインターネットゲートウェイにS3は対応していませんでしたが、 2022年より、インターフェースゲートポイント経由でルーティングしてS3バケットにアクセスすることもできるようになりました。

    Amazon CloudWatch Logs を使用して、Amazon Elastic Compute Cloud (Amazon EC2) インスタンス、 AWS CloudTrail、Route 53 およびその他のソースからログファイルをモニタリング、保存、およびアクセスすることができます。

    Q15#

    このシナリオでは、工場のデバイスから送信されるストリームデータを取得して保存するソリューションが必要です。 追加の要件としては、「ストリーミングデータはリアルタイムに利用されるわけではないため、保存に数分のタイムラグが発生しても問題ありませんが、 確実に保存される必要がないこと」、「コスト最適に、取得後10日以内のデータはすぐに利用できるようにして、 10日経過したデータはアーカイブすること」が求められています。

    工場のデバイスから送信されるストリームデータを取得して保存するソリューションとしては、 Amazon Kinesis Data Streamsを作成してデータストリームを取得して、Amazon S3のデータ保存します。 今回のシナリオでは、ストリームデータのリアルタイムにデータを取得して保存することになるため、 Amazon Kinesis Data Firehoseで直接データを取得して、Amazon S3バケットに保存する構成とします。

    Amazon Kinesis Data Streamsは1秒以下のデータロードが可能なように設計されており、ストリーミングデータを リアルタイムに処理したり、表示させたい場合に利用します。一方ではKinesis Data Firehoseはデータロードまで60秒必要となるため、 ミリ秒単位の処理には利用できませんが、データを配信して変換して保存する用途には向いています。
