    公式模擬試験#

    AWS Certified Solutions Architect – Associate Official Practice Question Set (SAA-C03 - Japanese)

    1#

    ある企業が Amazon EC2 ベースの MariaDB データベースを Amazon RDS に移行しようとしています。 同社は、同社の CPU とメモリの要件を満たすデータベースインスタンスタイプを既に特定しています。 データベースは、少なくとも 40 GiB のストレージ容量と 1,000 IOPS を提供する必要があります。

    最もコスト効率が良いのはどの Amazon RDS for MariaDB インスタンスのストレージ設定ですか。

    A: RDS インスタンス用に 350 GiB のマグネティックストレージをプロビジョンする。

    B: RDS インスタンス用に 50 GiB の汎用 SSD ストレージをプロビジョンする。

    C: RDS インスタンス用に 334 GiB の汎用 SSD ストレージをプロビジョンする。

    D: RDS インスタンス用に 1,000 IOPS で 50 GiB のプロビジョンド IOPS ストレージをプロビジョンする。

    解説#

    A: マグネティックストレージは、設定可能なパラメータとして IOPS をサポートしていません。

    B: 不正解です。汎用 SSD ストレージのベースライン I/O パフォーマンスは、GiB あたり 3 IOPS で、最低 100 IOPS です。 50 GiB のストレージの場合、ベースラインパフォーマンスは 150 IOPS になります。

    C: 正解です。汎用 SSD ストレージのベースライン I/O パフォーマンスは、GiB あたり 3 IOPS です。334 GiB のストレージの場合、 ベースラインパフォーマンスは 1,002 IOPS になります。さらに、汎用 SSD ストレージは、プロビジョンド IOPS ストレージよりもコスト効率に優れています。汎用 SSD ストレージの詳細については、「汎用 SSD ストレージ」を参照してください。

    D: 不正解です。1,000 IOPS の 50 GiB のプロビジョンド IOPS ストレージは、334 GiB の汎用 SSD ストレージよりもコストが高くなります。

    Info

    IOPSとは、ハードディスクドライブ（HDD）、ソリッドステートドライブ（SSD）、ストレージエリアネットワーク（SAN）などの コンピューターストレージデバイスの性能指標の一つで、1秒間に読み書きできる回数のこと。 IOPSの値が高いほど速くデータの読み書きができることを表しています。

    2#

    ある企業は、アプリケーションレイヤーとオンライントランザクション処理 (OLTP) リレーショナルデータベースで構成される 新しいアプリケーションを導入しています。 アプリケーションは常に利用可能でなければなりません。ただし、アプリケーションには非アクティブ期間があります。 同社は、こういったアイドル期間中のコンピューティングコストを最小限の金額にしたいと考えています。

    これらの要件を満たす最もコスト効率に優れたソリューションはどれですか。

    A: バースト可能なインスタンスタイプを使用して、Amazon EC2 インスタンスでアプリケーションを実行する。データベースには Amazon Redshift を使用する。

    B: AWS CloudFormation を使用して、アプリケーションと MySQL データベースを Amazon EC2 インスタンスにデプロイする。 アイドル期間の開始時にスタックを削除する。

    C: Application Load Balancer の背後にある Auto Scaling グループ内の Amazon EC2 インスタンスにアプリケーションをデプロイする。 データベースには Amazon RDS for MySQL を使用する。

    D: AWS Fargate で Amazon Elastic Container Service (Amazon ECS) を使用して、コンテナでアプリケーションを実行する。 データベースには Amazon Aurora サーバーレスを使用する。

    解説#

    A: 不正解です。EC2 バースタブルインスタンスは、スケーリングなしでバースト可能な機能を提供します。 ただし、このソリューションは、非アクティブ期間中のコストを最小限に抑えるものではなく、最もコスト効率に優れた選択肢ではありません。 さらに、Amazon Redshift データベースは OLTP には適していません。Amazon Redshift は、オンライン分析処理 (OLAP) 用に特別に設計されています。

    B: 不正解です。Infrastructure as Code (IaC) は可用性に役立ちますが、このソリューションは常時利用可能であるという要件を満たしていません。 さらに、このソリューションでは、データベースの再作成後にデータベースデータを保持する方法はありません。

    C: 不正解です。このソリューションでは、非アクティブ期間に少なくとも 1 つのインスタンスとデータベースが実行されます。 このソリューションでは、非アクティブ期間中のコストは最小限に抑えられません。このソリューションは、最もコスト効率に優れた選択肢ではありません。

    D: 正解です。Amazon ECS がコンピューティングに Fargate を使用する場合、アプリケーションがアイドル状態のときにコストは発生しません。 Aurora サーバーレスは、アイドル状態のときにコンピューティングコストも発生しません。Fargate の詳細については、 「AWS Fargate の料金」を参照してください。

    3#

    ある企業は、Linux ベースの Amazon EC2 インスタンスがどのように起動されたかに関する設定の詳細を調べる必要があります。

    システムメタデータを収集するために、ソリューションアーキテクトが EC2 インスタンスで実行する必要があるのはどのコマンドですか

    A: curl http://169.254.169.254/latest/meta-data

    B: curl http://localhost/latest/meta-data

    C: curl http://254.169.254.169/latest/meta-data

    D: curl http://192.168.0.1/latest/meta-data

    解説#

    A: 正解です。インスタンスメタデータを取得する唯一の方法は、リンクローカルアドレス (169.254.169.254) を使用することです。ンスタンスメタデータの詳細については、インスタンスメタデータの取得を参照してください。

    B: 不正解です。このソリューションでは 127.0.0.1 の IP アドレスがチェックされるため、localhost の使用は機能しません。メタデータは localhost 名を使用しても利用できません。

    C: 不正解です。リンクローカルアドレスの形式は 169.254.169.254 です。

    D: 不正解です。192.168.x.x.IP アドレスの範囲はパブリックブロックです。インスタンスメタデータは、パブリックブロックでは使用できません。

    4#

    衛星画像処理を行うある企業には、AWS で実行されるアプリケーションがあります。 同社は画像を Amazon S3 バケットに保存します。コンプライアンス上の理由から、 この企業はすべてのデータを月に 1 回オンプレミスの場所にレプリケートする必要があります。 同社が転送する必要があるデータの平均量は 60 TB です。

    このデータを転送するのに最もコスト効率に優れた方法はどれですか

    A: 既存の S3 バケットから Snowball Edge Storage Optimized デバイスに毎月データをエクスポートする。 デバイスをオンプレミスの場所に発送する。データを転送する。1 週間後にデバイスを返却する。

    B: S3 バケットレプリケーションを使用して、S3 Standard-Infrequent Access (S3 標準 – IA) ストレージを使用する新しい S3 バケットにすべてのオブジェクトをコピーする。AWS Storage Gateway ファイルゲートウェイを使用して、 新しい S3 バケットからオンプレミスの場所にデータを転送する。データの転送後、新しい S3 バケットから画像を削除する。

    C: S3 バケットレプリケーションを使用して、S3 Standard-Infrequent Access (S3 標準 – IA) ストレージを使用する新しい S3 バケットにすべてのオブジェクトをコピーする。Amazon S3 を使用して、 新しい S3 バケットからオンプレミスの場所にデータを転送する。データの転送後、新しい S3 バケットから画像を削除する。

    D: 既存の S3 バケット内のオブジェクトの Amazon CloudFront ディストリビューションを作成する。 CloudFront からオンプレミスの場所にオブジェクトを毎月ダウンロードする。

    解説#

    A: 正解です。基本料金は、デバイスとオンプレミスの場所で 10 日間の使用量をカバーします。 この企業が 1 週間以内にデバイスを返却した場合、支払うのは AWS からのデータ転送の基本価格と料金となります。 参照：AWS Snowball の料金

    B: 不正解です。S3 標準 – IA ストレージを使用する別の S3 バケットにすべてのデータをコピーする場合、 コスト面でのメリットはありません。元の S3 バケットから直接データを転送することも可能です。 このソリューションは、レプリケーションを追加するとコストが増加するため、最もコスト効率に優れた選択肢ではありません。 参照: AWS Storage Gateway の料金

    C: 不正解です。S3 標準 – IA ストレージを使用する別の S3 バケットにすべてのデータをコピーする場合、 コスト面でのメリットはありません。元の S3 バケットから直接データを転送することも可能です。 このソリューションは、レプリケーションを追加するとコストが増加するため、最もコスト効率に優れた選択肢ではありません。 参照:Amazon S3 の料金

    D: 不正解です。CloudFront へのデータ転送は無料ですが、CloudFront からオンプレミスの場所に 60 TB のデータを転送するにはコストがかかります。 このオプションは、Snowball Edge Storage Optimized デバイスを使用するオプションの約 2 倍のコストがかかります。 参照：Amazon CloudFront の料金

    5#

    ある企業に、EC2 Auto Scaling グループの一部である大規模な汎用 Amazon EC2 インスタンスタイプで実行されるアプリケーションがあります。 同社は、このアプリケーションに関連する将来的なコストを削減したいと考えています。同社が Amazon CloudWatch のメトリクスとログを確認したところ、 このアプリケーションが 1 日に 2、3 回ランダムに実行されてデータを取得、管理していることに気付きました。CloudWatch によると、 各リクエストの最大ランタイムは 10 分、メモリ使用量は 4 GB で、インスタンスは常に実行状態です。

    最もコストを削減できるのはどのソリューションですか。

    A: アプリケーションを大規模でバースト可能な EC2 インスタンスにデプロイする。

    B: アプリケーションコードを AWS Lambda 関数として実行するようにリファクタリングする

    C: Amazon Elastic Kubernetes Service (Amazon EKS) を使用してアプリケーションをコンテナ化する。EC2 インスタンスでコンテナをホストする。

    D: AWS Instance Scheduler を使用して、ログ内のランタイムに基づいてインスタンスを起動および停止する。

    解説#

    A: 不正解です。このソリューションでは、EC2 インスタンスが 1 日 24 時間稼働状態に保たれます。 このソリューションではコストは削減されません。参照: Amazon EC2 オンデマンド料金

    B: 正解です。Lambda はフルマネージドサービスであり、必要に応じて自動的にスケールし、高可用性を実現します。 アプリケーションを Lambda 関数としてデプロイするソリューションは、コストを最も削減します。参照:What is AWS Lambda?

    C: 不正解です。このソリューションでは、各リクエストの実行時間外も EC2 インスタンスが稼働状態になくてはならないため、 コストは削減されません。参照: Amazon Elastic Kubernetes Service

    D: 不正解です。リクエスト時間が予測できないため、このソリューションは適切ではありません。 このソリューションでは、インスタンスの起動が正しくスケジュールされていないためにアプリケーションが使用できないシナリオが作成されます。 このソリューションでは、正確なランタイムではなく、EC2 インスタンスに対して時間単位で課金されるため、不必要なコストも発生します。参照: AWS での Instance Scheduler

    6#

    企業には厳しいデータ保護要件があります。ソリューションアーキテクトは、VPC のセキュリティを設定して、 バックエンドの Amazon RDS DB インスタンスがインターネットからアクセスできないようにする必要があります。

    ソリューションアーキテクトは、DB インスタンスがアプリケーション層から特定のポートを介してのみアクセス可能であることを確認する必要があります。

    これらの要件を満たすために、このソリューションアーキテクトはどのようなアクションをとるべきですか (2 つ選択)

    A: DB インスタンスのプライベートサブネットのみを含む DB サブネットグループを指定する。

    B: プライベート IPv4 アドレスを指定した Elastic Network Interface を各 DB インスタンスにアタッチする。

    C: VPC で AWS Shield を設定する。DB インスタンスが使用するサブネットのルートテーブルを更新する。

    D: アプリケーション層とバックエンドの間のデータベースポートで AWS Direct Connect 接続を設定する。

    E: データベースポート経由でアプリケーション層のセキュリティグループからのリクエストを許可するインバウンドルールをデータベースセキュリティグループに追加する。他のインバウンドルールを削除する。

    解説#

    A: 正解です。プライベートサブネットは、データベース層の保護に使用する 1 つのコンポーネントです。 インターネットトラフィックはプライベートサブネットにルーティングされません。 DB インスタンスをプライベートサブネットに配置すると、セキュリティのレイヤーが追加されます。

    B: 不正解です。Elastic Network Interface は、仮想ネットワークカードを表す VPC 内の論理ネットワーキングコンポーネントです。 Elastic Network Interface の使用は、このシナリオの要件を満たしません。

    C: 不正解です。Shield は DDoS 攻撃からの保護を提供します。Shield をルートテーブル内のルートのターゲットにすることはできません。 Shield の使用は、シナリオの要件を満たしません。

    D: 不正解です。Direct Connect は、AWS 環境への専用接続を提供します。Direct Connect の使用は、このシナリオの要件を満たしません。

    E: 正解です。セキュリティグループは DB インスタンスへのアクセスを制限できます。 セキュリティグループは、特定のポート上のアプリケーション層からのみアクセスを提供します。

    アプリケーションは、ネットワークロードバランサーの背後にある 2 つの Amazon EC2 インスタンスで実行されます。 EC2 インスタンスは単一のアベイラビリティーゾーンにあります。

    このアーキテクチャの可用性を高めるために、ソリューションアーキテクトがするべきことはどれですか

    A: 元の EC2 インスタンスと同じアベイラビリティーゾーンに 2 つの新しい EC2 インスタンスを持つ新しい VPC を作成する。 2 つの VPC 間に VPC ピアリング接続を作成する。

    B: ネットワークロードバランサーを、Auto Scaling グループの EC2 インスタンスで設定された Application Load Balancer に置き換える。

    C: Amazon Route 53 を設定して、ネットワークロードバランサーの背後にある EC2 インスタンスでヘルスチェックを実行する。 フェイルオーバールーティングポリシーを追加する。

    D: EC2 インスタンスを、複数のアベイラビリティーゾーンにまたがる Auto Scaling グループに配置する。 Auto Scaling グループをネットワークロードバランサーのターゲットとして指定する。

    A: 不正解です。VPC ピアリングは、他のアベイラビリティーゾーンへ接続できるようにします。 ただし、EC2 インスタンスは 1 つのアベイラビリティーゾーンにまだ存在するため、VPC ピアリングでは高可用性は保証されません。

    B: 不正解です。ネットワークロードバランサーを Application Load Balancer に置き換えても、追加の可用性は提供されません。 両方のロードバランサーは本質的に高可用性です。ただし、EC2 インスタンスは 2 つのアベイラビリティーゾーンにまたがって拡張されている場合にのみ高可用性になります。

    C: 不正解です。フェイルオーバールーティングには、メインの宛先とセカンダリの(フェイルオーバー) 宛先が必要です。 このソリューションでは、フェイルオーバー先は指定されていません。さらに、EC2 インスタンスはまだ 1 つのアベイラビリティーゾーンにあるため、 この方法では高可用性は保証されません。

    D: 正解です。このソリューションは、EC2 インスタンスを複数のアベイラビリティーゾーンに拡張し、追加の容量が必要になったときに自動的に容量を追加します。 参考：Amazon EC2 Auto Scaling benefits

    C:不正解です。Aurora はリレーショナルデータベースです (key-value データベースではありません)。 Aurora は、ElastiCache を使用しても、マイクロ秒のレイテンシーを一貫して達成できる可能性は低いです。

    D:不正解です。Neptune は、高度に接続されたデータの操作に最適化されたグラフデータベースです。 Neptune は、単純なキーバリューデータ用に最適化されていません。

    ある企業が新しい Amazon EC2 インスタンスに新しいデータベースをデプロイしています。 このデータベースのワークロードには、最大 20,000 IOPS をサポートできる単一の Amazon Elastic Block Store (Amazon EBS) ボリュームが必要です。

    この要件を満たす EBS ボリュームのタイプはどれですか

    A: スループット最適化 HDD

    B: プロビジョンド IOPS SSD

    C: 汎用 SSD

    C:不正解です。汎用 SSD EBS ボリュームは、ボリュームごとに 16,000 IOPS に制限されています。

    D:不正解です。Cold HDD ボリュームは、IOPS ではなくスループットでパフォーマンスを示す、低コストの磁気ストレージに使用できます。 Cold HDD ボリュームは、大規模でシーケンシャルなコールドデータのワークロードに適しています。

    ある企業では、データレコードを最低 5 年間保持する必要があります。このデータは、保存後、アクセスされることはほとんどありません。 データは、2 時間以内にアクセスできるようにしておく必要があります。

    これらの要件を満たす、最もコスト効率に優れたソリューションはどれですか。

    A: データを Amazon Elastic File System (Amazon EFS) ファイルシステムに保存する。AWS Direct Connect を使用してデータにアクセスする。

    B: Amazon Elastic Block Store (Amazon EBS) ボリュームにデータを保存する。スナップショットを作成する。Amazon S3 バケットにスナップショットを保存する。

    C: データを Amazon S3 バケットに保存する。S3 ライフサイクルポリシーを使用して、S3 Standard-Infrequent Access (S3 標準 – IA) にデータを移動する。

    A: 不正解です。このソリューションは、最もコスト効率に優れたソリューションではありません。ファイルシステムの要件がない場合、Amazon S3 は、よりコスト効率に優れたソリューションです。

    B: 不正解です。このソリューションは、Amazon S3 に加えて Amazon EBS を使用する必要があるため、最もコスト効率に優れたソリューションではありません。

    C: 不正解です。S3 バケットにデータを保存すると、コスト効率に優れた初期のデータ保存場所となります。ただし、S3 標準 – IA は、シナリオの要件を満たすのに最もコスト効率に優れたストレージクラスではありません。S3 Glacier ストレージクラスは、低コストのデータアーカイブ用に設計されています。

    D: - 正解です。S3 バケットにデータを保存すると、コスト効率に優れた初期のデータ保存場所となります。S3 Glacier Instant Retrieval は、取得時間が 2 時間という要件を満たす、最もコスト効率に優れたアーカイブストレージソリューションです。 - S3 ストレージクラス間でデータを自動的に移動する方法の詳細については、「ストレージライフサイクルの管理」を参照してください。 - S3 ストレージクラスの詳細については、「Amazon S3 ストレージクラスを使用する」を参照してください。

    ある企業が、Amazon Elastic Container Registry (Amazon ECR) に保存している Amazon EC2 インスタンスおよびコンテナイメージの脆弱性スキャンを管理するためのサービスを調査しています。 このサービスは、ソフトウェアの潜在的な脆弱性を特定し、脆弱性の重要度を分類する必要があります。

    B: AWS Systems Manager の機能である Patch Manager

    C: Amazon Inspector

    B: 不正解です。Patch Manager は、オペレーティングシステムとアプリケーションにパッチを適用するために使用できます。これは、EC2 インスタンスとコンテナの脆弱性をスキャンするという要件を満たしていません。

    ある企業は、AWS の重要なアプリケーションの災害対策 (DR) アーキテクチャを設計しています。 同社は、AWS DR サイトのアプリケーションをサポートする最小インスタンス容量での RTO が 5 分であると判断しました。 同社は DR アーキテクチャのコストを最小限に抑える必要があります。

    これらの要件を満たす DR 戦略はどれですか。

    A: ウォームスタンバイ

    B: 不正解です。このソリューションは、RTO 5 分の要件を満たしません。インスタンスはアイドル状態であり、 5 分以内に実行できず、フルキャパシティーまでスケールできません。

    C: 不正解です。これは Active-Active 環境であるため、このソリューションは RTO の要件に瞬時に対応します。 その時間内に、サービスは既にフルキャパシティーで稼働しています。ただし、このソリューションには、 同社の要件を満たすために必要以上のコストがかかります。

    AWS で Site-to-Site VPN 接続を構築するために必要なコンポーネントはどれですか (2 つ選択)

    A: インターネットゲートウェイ

    B: NAT ゲートウェイ

    C: カスタマーゲートウェイ

    D: Amazon API Gateway

    E: 仮想プライベートゲートウェイ

    解説#

    A: 不正解です。インターネットゲートウェイは VPC にアタッチされ、インターネットからのトラフィックが VPC に出入りできるようにします。VPN 接続は、インターネットゲートウェイを経由しません。インターネットゲートウェイは、暗号化された VPN 接続ではなく、オープンインターネットからのトラフィックを許可するように設計されています。

    B: 不正解です。NAT ゲートウェイは、プライベート Amazon EC2 インスタンスがインターネットにリクエストを送信する方法を提供します。NAT ゲートウェイでは、暗号化された Site-to-Site VPN 接続を作成することはできません。

    C: 正解です。VPN 接続を確立するには、カスタマーゲートウェイが必要です。カスタマーゲートウェイデバイスは、お客様のデータセンターでセットアップされ、設定されます。

    D: 不正解です。API Gateway は、あらゆる規模で API を作成、配布、保守、モニタリング、保護できる、デベロッパー向けのフルマネージドサービスです。API は、バックエンドサービスからデータ、ビジネスロジック、または機能にアクセスするために使用するアプリケーションのフロントドアとして機能します。ただし、API Gateway は VPN 接続の実装には必要ありません。

    E: 正解です。仮想プライベートゲートウェイは VPC にアタッチされ、AWS で Site-to-Site VPN 接続を作成します。オープンなパブリックインターネットを経由することなく、オンプレミスのデータセンターから VPC へのプライベート暗号化ネットワークトラフィックを受け入れることができます。

    ある企業は、ソリューションアーキテクトに、既存のオンプレミスアプリケーションに対してパイロットライトを使用した災害対策 (DR) 戦略を実装するよう依頼しています。

    アプリケーションは自己完結型で、データベースにアクセスする必要はありません。

    パイロットライトの DR 戦略を実装するソリューションはどれですか。

    A: オンプレミスのアプリケーション、設定、データを Amazon S3 バケットにバックアップする。オンプレミスのアプリケーションで障害が発生した場合は、AWS で新しいホスティング環境を構築し、S3 バケットに保存されている情報からアプリケーションを復元する。

    B: Amazon EC2 インスタンスを使用して AWS でアプリケーションホスティング環境を再作成し、EC2 インスタンスを停止する。オンプレミスのアプリケーションに障害が発生した場合、停止した EC2 インスタンスを起動し、アプリケーショントラフィックの 100% を AWS クラウドで実行されている EC2 インスタンスに転送する。

    C: Amazon EC2 インスタンスを使用して、AWS でアプリケーションホスティング環境を再作成する。アプリケーショントラフィックの 10% を、AWS クラウドで実行されている EC2 インスタンスに転送する。オンプレミスのアプリケーションに障害が発生した場合、アプリケーショントラフィックの 100% を AWS クラウドで実行されている EC2 インスタンスに転送する。

    D: オンプレミスのアプリケーション、設定、データを Amazon S3 バケットにバックアップする。オンプレミスのアプリケーションに障害が発生した場合、オンプレミスのホスティング環境を再構築し、S3 バケットに保存されている情報からアプリケーションを復元する。

    解説#

    A: 不正解です。これはバックアップと復元を使用した DR 戦略です。バックアップと復元を使用した DR 戦略は通常、コストが最も低いものの、リカバリ時間が最も長くなります。AWS でホスティングインフラストラクチャを手動で再構築するソリューションには、数時間かかる場合があります。

    B: 正解です。これはパイロットライトを使用した DR 戦略です。このソリューションは、AWS リージョンに既存のアプリケーションホスティング環境を再作成します。 このソリューションは、ほとんど (またはすべて) のリソースをオフにし、テスト中または DR フェイルオーバーが必要な場合にのみリソースを使用します。 RPO と RTO は、通常数十分です。パイロットライト戦略は、コアインフラストラクチャの要件がすべて揃っているため、災害時の復旧を簡素化します。 パイロットライト戦略は、アクティブなリソースを最小限に抑えることで、DR の継続的なコストも最小限に抑えます。

    パイロットライトの DR 戦略の詳細については、「Disaster recovery options in the Cloud」(クラウドの災害対策オプション) を参照してください。

    C:不正解です。これはウォームスタンバイを使用した DR 戦略です。このソリューションは、AWS リージョンに既存のアプリケーションホスティング環境を再作成します。 このソリューションは、ライブトラフィックの一部を処理します。この DR 戦略では、RPO と RTOは通常数分です。ただし、このソリューションはリソースを継続的に稼働させるため、 コストが高くなります。

    D: 不正解です。これはバックアップと復元を使用した DR 戦略です。バックアップと復元を使用した DR 戦略は通常、 コストが最も低いものの、リカバリ時間が最も長くなります。オンプレミスのホスティングインフラストラクチャを手動で再構築し、 企業がバックアップしたデータをダウンロードするソリューションには、数時間から数日かかる場合があります。

    ある企業では、本番稼働用ワークロードを実行するのに、1 つの AWS アカウントを使用しています。 同社には、セキュリティチーム用に別の AWS アカウントがあります。 定期的な監査では、セキュリティチームは、本番稼働用ワークロードを実行する AWS アカウントの特定のアカウント設定とリソース設定を表示する必要があります。 ソリューションアーキテクトは、AWS セキュリティのベストプラクティスに従ったソリューションを設計することにより、セキュリティチームに必要なアクセスを提供する必要があります。

    A: 本番稼働用アカウントのセキュリティチームメンバーごとに IAM ユーザーを作成する。 セキュリティチームが必要とするアクセス権限を各ユーザーに提供する許可ポリシーをアタッチする。

    D: 本番稼働用アカウントのセキュリティチームメンバーごとに IAM ユーザーを作成する。 セキュリティチームが必要とするアクセス権限を提供する許可ポリシーを新しい IAM グループにアタッチする。 セキュリティチームメンバーをグループに割り当てる。

    解説#

    A: 不正解です。このソリューションは、ロールを使用してアクセス許可を委任するセキュリティのベストプラクティスに従っていません。

    B: 正解です。このソリューションは、ロールを使用して最小特権アクセスで構成されるアクセス許可を委任することにより、 セキュリティのベストプラクティスに従います。ロールを使用してアクセス許可を委任する方法の詳細については「ロールを使用してアクセス許可を委任する」を参照してください。

    C: 不正解です。ユーザーへの管理者権限の割り当ては、セキュリティのベストプラクティスおよび最小権限の原則に違反します。

    D: 不正解です。このソリューションは、ロールを使用してアクセス許可を委任するセキュリティのベストプラクティスに従っていません。

    あるレポートアプリケーションは、Application Load Balancer (ALB) の背後にある Amazon EC2 インスタンスで実行されます。 このインスタンスは、複数のアベイラビリティーゾーンにまたがる Amazon EC2 Auto Scaling グループ内で実行されます。 複雑なレポートの場合、アプリケーションがリクエストに応答するまでに最大で 15 分かかることがあります。

    ソリューションアーキテクトは、スケールインイベント中にレポートリクエストが処理中である場合、ユーザーに HTTP 5xx エラーが出されることを懸念しています。

    インスタンスが終了する前にユーザーリクエストを確実に完了させるために、このソリューションアーキテクトがするべきことはどれですか

    A: インスタンスのターゲットグループのスティッキーセッション (セッションアフィニティ) を有効にする。

    B: Application Load Balancer ターゲットグループのインスタンスサイズを増やす。

    C: Auto Scaling グループのクールダウン期間を、実行時間が応答に必要な最長時間よりも長くする。

    D: インスタンスのターゲットグループに対する登録解除遅延のタイムアウトを 900 秒以上に増やす。

    解説#

    A: 不正解です。スケールインプロセス中に EC2 インスタンスがターゲットグループから削除された場合、EC2 インスタンスは失敗します (または、チェックされると異常になります)。Application Load Balancer は、そのターゲットへのリクエストのルーティングを停止し、新しい正常なターゲットを選択します。

    B: 不正解です。インスタンスサイズを増やすと、処理速度が上がる可能性があります。ただし、このソリューションでは、リクエストを処理するインスタンスがスケールインアクションの影響を受けないことを直接保証するものではありません。

    C: 不正解です。Amazon EC2 Auto Scaling のクールダウン期間は、以前のアクティビティの影響が明らかになる前に、Auto Scaling グループが追加のインスタンスを起動または終了するのを防ぐのに役立ちます。

    D: 正解です。デフォルトでは、Elastic Load Balancing 登録解除プロセスが完了するまで 300 秒待って、ターゲットへ処理中のリクエストが完了するのを助けることができます。Elastic Load Balancing の待機時間を変更するには、登録解除の遅延値を更新します。

    17#

    ある企業には、Web サイトのユーザーに関するログファイルをエクスポートするオンプレミスアプリケーションがあります。 これらのログファイルのサイズは 20 GB から 30 GB までです。ソリューションアーキテクトは、これらのファイルを保存するための Amazon S3 バケットを作成しました。 ファイルはアプリケーションから直接アップロードされます。ネットワーク接続に断続的な障害が発生し、アップロードが失敗することがあります。

    ソリューションアーキテクトは、この問題を解決するソリューションを設計する必要があります。 このソリューションでは、オペレーションのオーバーヘッドを最小限に抑える必要があります。

    これらの要件を満たすソリューションはどれですか

    A: S3 Transfer Acceleration を有効にする。

    B: 最も近い AWS リージョンの Amazon EC2 インスタンスにファイルをコピーする。 S3 ライフサイクルポリシーを使用して、ログファイルを Amazon S3 にコピーする。

    C: Amazon S3 へのマルチパートアップロードを使用する。

    D: ファイルを 2 つの AWS リージョンに同時にアップロードする。 2 つのリージョン間で双方向クロスリージョンレプリケーションを有効にする。

    解説#

    A: 不正解です。S3 Transfer Acceleration は、エッジロケーションを使用してデータを Amazon S3 にコピーすることで、 より迅速なアップロードを容易にします。S3 Transfer Acceleration では、1 回の PUT オペレーションのファイルサイズ制限 (5 GB) の問題は解決されません。

    B: 不正解です。このソリューションでは、1 回の PUT オペレーションのファイルサイズ制限(5 GB) の問題は解決しません。 S3 ライフサイクルポリシーは、EC2 ブロックストレージから Amazon S3 にファイルを転送することはできません。 また、このソリューションにより、環境に不要なサービスやオペレーションのオーバーヘッドが追加されます。

    C: 正解です。1 回の PUT オペレーションで、最大 5 GB のサイズの 1 つのオブジェクトをアップロードできます。 マルチパートアップロードを使用すると、このシナリオのファイルなど、より大きなファイルをアップロードできます。

    D: 不正解です。このソリューションでは、1 回の PUT オペレーションのファイルサイズ制限（5 GB）の問題は解決しません。 転送先の各リージョンは、1 つのリージョンと同じ問題を抱えています。このソリューションでは、オペレーションのオーバーヘッドも増加します。

    Note

    マルチパートアップロードの詳細については、「マルチパートアップロードを使用したオブジェクトのアップロードとコピー」を参照してください。

    18#

    ある企業は、Amazon CloudFront ディストリビューションのオリジンとして設定された Application Load Balancer の背後にある Amazon EC2 インスタンスでウェブサイトを運営しています。同社は、クロスサイトスクリプティングと SQL インジェクション攻撃から保護したいと考えています。

    これらの要件を満たすために、ソリューションアーキテクトが推奨すべきアプローチはどれですか。

    A: AWS Shield Advanced を有効にする。CloudFront ディストリビューションを保護されたリソースとしてリストする。

    B: AWS Firewall Manager で AWS Shield Advanced のポリシーを定義して、クロスサイトスクリプティングと SQL インジェクション攻撃をブロックする。

    C: EC2 インスタンスに AWS Firewall Manager をデプロイする。クロスサイトスクリプティングと SQL インジェクション攻撃をブロックする条件とルールを作成する。

    D: CloudFront ディストリビューションで AWS WAF をセットアップする。クロスサイトスクリプティングと SQL インジェクション攻撃をブロックする条件とルールを使用する。

    解説#

    A: 不正解です。Shield Advanced は DDoS 攻撃から保護します。Shield Advanced は、クロスサイトスクリプティングや SQL インジェクションからは保護しません。

    B: 不正解です。Firewall Manager を使用すると、AWS WAF、Shield Advanced、その他の AWS サービスを管理できます。 Shield Advanced は DDoS 攻撃から保護します。Shield Advanced は、クロスサイトスクリプティングや SQL インジェクションからは保護しません。

    C: 不正解です。Firewall Managerを使用すると、AWS WAF、AWS Shield Advanced、およびその他の AWS サービスを管理できます。 Firewall Manager は、EC2 インスタンスにインストールされていないマネージド型サービスです。

    D: 正解です。AWS WAF は、悪意のある可能性のある SQL コード (SQL インジェクション) の存在を検出できます。 AWS WAF は、悪意のある可能性のあるスクリプト (クロスサイトスクリプティング) の存在を検出することもできます。 参照：AWS WAF

    19#

    ある企業が、AWS Cloud で人気の高いウェブアプリケーションの新しいモバイルバージョンを開発しています。 このモバイルアプリケーションは、内部および外部のユーザーがアクセスできる必要があります。 そして、承認、認証、およびユーザー管理を 1 つの中央ソースから処理する必要があります。

    これらの要件を満たすのはどのソリューションですか

    A: IAM ロール

    B: IAM ユーザーとグループ

    C: Amazon Cognito ユーザープール

    D: AWS Security Token Service (AWS STS)

    解説#

    A: 不正解です。IAM ロールは、IAM ユーザーによって引き受けられる IAM エンティティです。 IAM ロールには、エンティティができることとできないことを定義するアクセス権限ポリシーがあります。 ただし、IAM ロールがアプリケーションへのアクセスを制御することはありません。

    B: 不正解です。IAM ユーザーとグループを使用して、AWS サービスの使用を認証および承認されるユーザーを制御できます。 ただし、ユーザーおよびグループがアプリケーションへのアクセスを制御することはありません。

    C: 正解です。Amazon Cognito は、ウェブアプリケーションやモバイルアプリケーションの認証、許可、ユーザー管理をサポートしています。 ユーザーは、ユーザー名とパスワードを使用して直接サインインすることも、信頼できるサードパーティーを通じてサインインすることもできます。 参照：What is Amazon Cognito?

    D: 不正解です。AWS STS を使用して、AWS リソースへのアクセスを制御できる一時的セキュリティ認証情報を持つ、 信頼されたユーザーを作成および提供することができます。ただし、AWS STS がアプリケーションへのアクセスを制御することはありません。

    20#

    あるメディア企業が、グラフィックレンダリングの新しいソリューションを設計しています。 このアプリケーションでは、フレームのレンダリング後に破棄される一時データ用に最大 400 GB のストレージが必要です。 このアプリケーションでは、レンダリングの実行に約 40,000 のランダム IOPS が必要です。

    このレンダリングアプリケーションで最もコスト効率に優れたストレージの選択肢はどれですか。

    A: インスタンスストアストレージを備えた、ストレージ最適化された Amazon EC2 インスタンス

    B: プロビジョンド IOPS SSD (io1 または io2) Amazon Elastic Block Store (Amazon EBS) ボリュームを持つ、ストレージ最適化された Amazon EC2 インスタンス

    C: スループット最適化された HDD (st1) Amazon Elastic Block Store (Amazon EBS) ボリュームを備えたバースト可能な Amazon EC2 インスタンス

    D: VPC エンドポイントを介した Amazon S3 ストレージを備えたバースト可能な Amazon EC2 インスタンス

    解説#

    A: 正解です。SSD バックドストレージ最適化 (i2) インスタンスは、365,000 を超えるランダム IOPS を提供します。 インスタンスストアには、インスタンスの通常の時間単位のコストと比較して、追加コストはありません。

    B: 不正解です。プロビジョンド IOPS SSD (io1 または io2) EBS ボリュームは、シナリオで必要な 40,000 IOPS 以上を提供できます。 ただし、Amazon EBS では時間単位のインスタンス料金にコストが加算されるため、このソリューションはインスタンスストアほどコスト効率に優れていません。 このソリューションは、インスタンスのライフサイクルを超えてデータの永続性を提供しますが、このユースケースに永続性は必要ありません。

    C: 不正解です。スループット最適化 HDD (st1) EBS ボリュームは、ランダム IOPS ではなく、 ボリュームとの間で送受信できるデータのスループットを最大化するように設計されています。 したがって、このソリューションは IOPS 要件を満たしていません。さらに、Amazon EBS では、時間単位のインスタンスレートにコストが加算されます。 このソリューションは、インスタンスのライフサイクルを超えてデータの永続性を提供しますが、このユースケースに永続性は必要ありません。

    D: 不正解です。スクラッチボリューム領域に必要なデータが急速に変化するため、Amazon S3 (オブジェクトストレージ) は間違ったストレージになります。 ブロックストレージは、読み込み/書き込み機能がスムーズに動作するのに適しています。

    Note

    • i2 ストレージの詳細については、「インスタンスストアボリューム」を参照してください。
    • EC2 インスタンスの料金の詳細については、「Amazon EC2 の料金」を参照してください。
